# ACTIVE SPEC: Phase 17.8 - Dynamic Player Type Cascading

## 1. Context & Objective
In Phase 16.8, we introduced Custom Player Types at the club settings level. However, renaming a player type (e.g., from "Performance" to "Premier") currently leaves existing players orphaned with the old tag, breaking filters on both the Coach UI and the RDO Master Dashboard (Phase 17.5).

**Critical Database Block:** The `players` table still has a hardcoded CHECK constraint:
```sql
CHECK (player_type IN ('Performance','Open','Women''s'))
```
This literally blocks the database from accepting "Premier" as a player_type value!

**Goal:** Create a database-level cascading update with safe batch processing. When a player type is renamed in `club_settings`, an RPC must automatically update all players in that club who currently hold the old type, and the frontend must dynamically pull filter options from the settings.

## 2. Current State Analysis

### Database Constraints
- ✅ **CHECK constraint exists**: `CHECK (player_type IN ('Performance','Open','Women''s'))` in `001_schema.sql`
- ✅ **Phase 16.8 incomplete**: Migration `022_` added `player_types` to `club_settings` but didn't drop the CHECK constraint

#### 1.2 Create Safe Cascade RPC with Batch Processing
```sql
-- Create RPC function with batch processing for large clubs
CREATE OR REPLACE FUNCTION rename_custom_player_type(
  p_club_id UUID,
  p_old_type TEXT,
  p_new_type TEXT,
  p_batch_size INT DEFAULT 50  -- Safe default for UI responsiveness
)
RETURNS JSONB
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
DECLARE
  total_players INT;
  updated_count INT := 0;
  batch_count INT := 0;
  batch_number INT := 1;
BEGIN
  -- Validate inputs
  IF p_old_type = p_new_type THEN
    RETURN jsonb_build_object(
      'success', false,
      'error', 'Old and new types are identical',
      'error_code', 'VALIDATION_ERROR'
    );
  END IF;
  
  -- Get total count of affected players
  SELECT COUNT(*) INTO total_players
  FROM players 
  WHERE club_id = p_club_id 
    AND player_type = p_old_type;
    
  -- Early return if no players to update
  IF total_players = 0 THEN
    -- Still update club_settings array
    UPDATE club_settings
    SET player_types = ARRAY(
      SELECT CASE WHEN t = p_old_type THEN p_new_type ELSE t END
      FROM unnest(player_types) AS t
    )
    WHERE club_id = p_club_id;
    
    RETURN jsonb_build_object(
      'success', true,
      'total_players', 0,
      'updated_players', 0,
      'batches', 0,
      'message', 'No players to update, updated club settings only'
    );
  END IF;
  
  -- Process in batches to prevent timeouts and lock contention
  WHILE updated_count < total_players LOOP
    -- Update one batch of players
    UPDATE players 
    SET player_type = p_new_type
    WHERE id IN (
      SELECT id 
      FROM players 
      WHERE club_id = p_club_id 
        AND player_type = p_old_type
      ORDER BY id  -- Consistent ordering
      LIMIT p_batch_size
      OFFSET updated_count
      FOR UPDATE SKIP LOCKED  -- Prevent deadlocks
    );
    
    -- Get number of rows updated in this batch
    GET DIAGNOSTICS batch_count = ROW_COUNT;
    updated_count := updated_count + batch_count;
    
    -- Small delay between batches to prevent lock contention
    -- Only if we have more batches to process
    IF updated_count < total_players THEN
      PERFORM pg_sleep(0.01); -- 10ms delay between batches
    END IF;
    
    batch_number := batch_number + 1;
  END LOOP;
  
  -- Update club_settings.player_types array (replace old with new)
  UPDATE club_settings
  SET player_types = ARRAY(
    SELECT CASE WHEN t = p_old_type THEN p_new_type ELSE t END
    FROM unnest(player_types) AS t
  )
  WHERE club_id = p_club_id;
  
  -- Return success with detailed statistics
  RETURN jsonb_build_object(
    'success', true,
    'total_players', total_players,
    'updated_players', updated_count,
    'batches', CEIL(total_players::FLOAT / p_batch_size),
    'batch_size', p_batch_size,
    'message', format('Successfully updated %s players in %s batches', 
                     total_players, CEIL(total_players::FLOAT / p_batch_size))
  );
EXCEPTION
  WHEN OTHERS THEN
    -- Rollback is automatic in PL/pgSQL functions
    RETURN jsonb_build_object(
      'success', false,
      'error', SQLERRM,
      'error_code', SQLSTATE,
      'updated_so_far', updated_count,
      'message', format('Failed at batch %s after updating %s players', 
                       batch_number, updated_count)
    );
END;
$$;

-- Grant execute permission to authenticated users
GRANT EXECUTE ON FUNCTION rename_custom_player_type TO authenticated;
```

#### 2.2 Update Player Type Edit Handler
```tsx
// In ClubSettings.tsx - update the updatePlayerType function
async function handlePlayerTypeRename(idx: number, newValue: string) {
  const oldValue = playerTypes[idx];
  if (oldValue === newValue) return;
  
  // Check if this is an existing type being renamed (not a new empty one)
  const isRenamingExistingType = oldValue.trim() !== '' && 
                                 clubSettings?.player_types?.includes(oldValue);
  
  if (isRenamingExistingType) {
    // Show confirmation modal with player count
    setRenameModal({
      show: true,
      oldType: oldValue,
      newType: newValue,
      index: idx
    });
  } else {
    // Just update the UI for new types or empty renames
    updatePlayerType(idx, newValue);
  }
}

// New state for rename modal
const [renameModal, setRenameModal] = useState<{
  show: boolean;
  oldType: string;
  newType: string;
  index: number;
} | null>(null);

// Function to execute the rename
async function executePlayerTypeRename() {
  if (!renameModal || !activeClubId) return;
  
  const { oldType, newType, index } = renameModal;
  setSaving(true);
  
  try {
    // Call the RPC with batch size 50
    const { data, error } = await supabase.rpc('rename_custom_player_type', {
      p_club_id: activeClubId,
      p_old_type: oldType,
      p_new_type: newType,
      p_batch_size: 50
    });
    
    if (error) throw error;
    
    if (data?.success) {
      // Update local state
      updatePlayerType(index, newType);
      toast.success(`Updated ${data.updated_players} players successfully`);
    } else {
      throw new Error(data?.error || 'Rename failed');
    }
  } catch (err) {
    toast.error(`Failed to rename: ${err.message}`);
    // Don't update UI on error
  } finally {
    setSaving(false);
    setRenameModal(null);
  }
}
```

## 8. Risk Mitigation

### Technical Risks
1. **Database Timeouts**: Mitigated by batch processing (50 players/batch)
2. **Lock Contention**: Mitigated by `SKIP LOCKED` and 10ms delays between batches
3. **Partial Updates**: Mitigated by RPC atomicity (all-or-nothing)
4. **UI Freeze**: Mitigated by async operations with progress feedback
5. **Network Failure**: Mitigated by detailed error reporting and retry options

### Business Risks
1. **Data Corruption**: RPC validates inputs and uses transactions
2. **User Confusion**: Clear modals explain what will happen
3. **Performance Impact**: Batch size 50 ensures UI remains responsive
4. **Multi-club Conflicts**: RPC is club-scoped, no cross-club contamination
