# COMPLETED SPEC: Phase 17.3 - God Mode Hydration & Data Safety

## Implementation Date: 2026-04-14
## Status: ✅ Complete

## 1. Context & Objective

Phase 17.3 addressed critical data safety and hydration issues in the RDO (Rugby Development Officer) "God Mode" feature introduced in Phases 17.1-17.2. When RDOs switch between clubs using the `switchTenant` function, React's data hooks could experience race conditions, stale data, and improper cleanup of realtime subscriptions.

**Primary Objective:** Implement robust hydration safety patterns to prevent data leaks, ensure clean component remounting, and protect against race conditions during tenant switching.

## 2. Problem Analysis

### Issues Identified
1. **Race Conditions**: Fast switching between clubs could cause hooks to fetch data with outdated `activeClubId`
2. **Stale Subscriptions**: Realtime subscriptions not properly cleaned up during tenant switches
3. **UI State Mismatch**: Component state persisting across tenant boundaries
4. **Memory Leaks**: Unsubscribed event listeners and realtime channels

### Root Causes
- React components not fully remounting when `activeClubId` changes
- Hooks continuing to execute with previous tenant context
- Lack of cancellation patterns for in-flight requests
- Realtime subscriptions not tied to current tenant context

## 3. Solution Architecture

### 3.1 React Tree Hard Remount
```typescript
// In App.tsx - Force complete React tree remount on tenant switch
<App key={activeClubId || 'command-center'}>
  {/* All application components */}
</App>
```

**Benefits:**
- Guarantees clean component state for each tenant
- Automatic cleanup of all hooks and subscriptions
- Eliminates race conditions from previous tenant data
- Simple and robust pattern

### 3.2 Enhanced SwitchTenant Function
```typescript
// In AuthContext.tsx - Added switching state and protection
const switchTenant = async (clubId: string | null) => {
  if (switching) return; // Prevent concurrent switches
  
  setSwitching(true);
  try {
    // Clear all cached data
    queryClient.clear();
    
    // Update auth state
    setAuthState(prev => ({
      ...prev,
      activeClubId: clubId,
      switching: false
    }));
    
    // Force UI refresh
    window.location.reload(); // Or React tree remount
  } catch (error) {
    console.error('Tenant switch failed:', error);
    setSwitching(false);
  }
};
```

### 3.3 Three-Layer Hook Protection
Implemented in all data hooks (`usePlayers`, `useWeeks`, `useSelectionBoard`, etc.):

**Layer 1: Active Club Guard**
```typescript
if (!activeClubId) {
  // Return empty/loading state for RDOs in command center
  return { data: [], loading: false, error: null };
}
```

**Layer 2: Ignore Flag Pattern**
```typescript
const [ignore, setIgnore] = useState(false);

useEffect(() => {
  if (ignore) return;
  
  // Fetch data only when not ignored
  fetchData();
  
  return () => {
    setIgnore(true); // Mark for ignore on unmount
  };
}, [activeClubId, ignore]);
```

**Layer 3: Realtime Subscription Safety**
```typescript
useEffect(() => {
  if (!activeClubId) return;
  
  const channel = supabase
    .channel(`players-${activeClubId}`)
    .on('postgres_changes', 
      { event: '*', schema: 'public', table: 'players', filter: `club_id=eq.${activeClubId}` },
      handleChange
    )
    .subscribe();
  
  return () => {
    supabase.removeChannel(channel); // Cleanup on unmount
  };
}, [activeClubId]); // Channel tied to specific club_id
```

## 8. Architecture Decisions Locked

1. **Hard Remount Pattern**: Using `key={activeClubId}` for guaranteed cleanup
2. **Three-Layer Hook Protection**: Defense-in-depth against race conditions
3. **Club-Scoped Realtime**: Subscriptions filtered by `club_id` for efficiency
4. **Switching State Guard**: Prevents concurrent tenant switches
5. **Command Center as Null State**: RDOs with `activeClubId=null` see different UI
