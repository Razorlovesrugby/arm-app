# ACTIVE SPEC: Phase 17.5 - RFC Player Pool & Master Grids

## 🎯 Context & Objective
The RDO (RFC Manager) needs to manage multiple teams (e.g., Prems, Women's, U85kg) where each team acts as a distinct `club_id` in the database. Our priority is to build out the master management screens on the `/rdo-dashboard`. The RDO must be able to view a master grid of all players across all teams and easily filter them, while clicking a player should only open a read-only summary panel.

**Goal:** Build the "RFC Player Pool" data grid with specific filters, powered by a highly optimized Supabase RPC function.

## 🏗️ Architecture Decisions

### 1. **Database Optimization Strategy**
**Option A: Supabase Database Function (RPC)**
- Create `get_rfc_player_pool()` PostgreSQL function
- Single database call, maximum performance
- Complex SQL logic hidden from frontend
- SECURITY DEFINER to respect RLS bypass from Phase 17.1

**Option B: Frontend Joins with Multiple Queries**
- Multiple Supabase queries with client-side joining
- Higher network overhead, N+1 query risk
- Easier to debug but poorer performance at scale

**Decision:** Implement **Option A** with RPC function. The RDO may manage 10+ clubs with 50+ players each (500+ total records). A single optimized function prevents frontend N+1 queries and provides best performance.

### 2. **RPC Function Security Model**
- Use `SECURITY DEFINER` with `SET search_path = public`
- Function executes with service role privileges but respects RLS
- Input parameter: `auth.uid()` passed from authenticated session
- Access control via `rdo_club_access` table lookup

### 3. **UI Component Structure**
- **New Page**: `RfcPlayerPool.tsx` as tab in RDO Layout
- **Grid Component**: Reuse existing table patterns from ReadinessMatrix
- **Filters Bar**: Above table with dropdowns/toggles
- **Read-Only Panel**: Reuse `PlayerFormSheet` with editing disabled

### 4. **Data Flow Design**
1. RDO navigates to Player Pool tab
2. Frontend calls `get_rfc_player_pool()` RPC with `auth.uid()`
3. Database returns joined dataset with player + team + availability
4. Frontend renders filtered grid
5. Click row → opens read-only slide-out with player details

## 🎨 Logic Implementation

### Step 1: Database Function (RPC) - `supabase/migrations/022_phase_17_5_rfc_pool.sql`

**Function Requirements:**
- Input: `auth.uid()` (UUID)
- Returns: Joined JSON/recordset containing:
  1. Player details: `id`, `first_name`, `last_name`, `position_primary`, `player_type`, `status`
  2. Team Name: `clubs.name`
  3. Current Week Availability: "Available", "Unavailable", "Pending", or NULL

**SQL Function Implementation:**
```sql
CREATE OR REPLACE FUNCTION get_rfc_player_pool(user_uuid UUID)
RETURNS TABLE (
  player_id UUID,
  first_name TEXT,
  last_name TEXT,
  position_primary TEXT,
  player_type TEXT,
  status TEXT,
  team_name TEXT,
  club_id UUID,
  current_availability TEXT
) 
SECURITY DEFINER
SET search_path = public
LANGUAGE plpgsql
AS $$
BEGIN
  RETURN QUERY
  SELECT 
    p.id AS player_id,
    p.first_name,
    p.last_name,
    p.position_primary,
    p.player_type,
    p.status,
    c.name AS team_name,
    p.club_id,
    COALESCE(
      CASE ar.status
        WHEN 'available' THEN 'Available'
        WHEN 'unavailable' THEN 'Unavailable'
        WHEN 'pending' THEN 'Pending'
        ELSE NULL
      END,
      'No Response'
    ) AS current_availability
  FROM players p
  JOIN clubs c ON p.club_id = c.id
  LEFT JOIN rdo_club_access rca ON c.id = rca.club_id
  LEFT JOIN weeks w ON c.id = w.club_id 
    AND w.is_active = true
  LEFT JOIN availability_responses ar ON p.id = ar.player_id 
    AND w.id = ar.week_id
  WHERE rca.user_id = user_uuid
    AND p.is_active = true
  ORDER BY c.name, p.last_name, p.first_name;
END;
$$;
```

### Step 4: Custom Hook (`src/hooks/useRFCPlayerPool.ts`)

**Hook Implementation:**
```typescript
export interface PlayerPoolRow {
  player_id: string;
  first_name: string;
  last_name: string;
  position_primary: string;
  player_type: string;
  status: string;
  team_name: string;
  club_id: string;
  current_availability: string;
}

export function useRFCPlayerPool() {
  const { user, role } = useAuth();
  const [data, setData] = useState<PlayerPoolRow[]>([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<string | null>(null);

  useEffect(() => {
    // Role guard
    if (role !== 'rdo' || !user) {
      setData([]);
      setLoading(false);
      return;
    }

    let ignore = false;

    async function fetchPlayerPool() {
      setLoading(true);
      setError(null);

      try {
        const { data: poolData, error: poolError } = await supabase
          .rpc('get_rfc_player_pool', { user_uuid: user.id });

        if (!ignore) {
          if (poolError) throw new Error(poolError.message);
          setData(poolData || []);
          setLoading(false);
        }
      } catch (err) {
        if (!ignore) {
          setError(err instanceof Error ? err.message : 'Failed to fetch player pool');
          setLoading(false);
        }
      }
    }

    fetchPlayerPool();

    return () => { ignore = true };
  }, [user, role]);

  return { data, loading, error };
}
```

## ⚠️ Risk Mitigation

### 1. **RPC Performance with Many Clubs**
- **Risk**: RPC could be slow for RDOs with 20+ clubs
- **Mitigation**: Add indexes on `rdo_club_access(user_id)`, `players(club_id, is_active)`
- **Monitoring**: Log RPC execution times in production
- **Fallback**: Implement pagination or virtual scrolling in UI

### 2. **Filter Complexity**
- **Risk**: Multiple combined filters could confuse users
- **Mitigation**: Clear visual feedback, filter badges, "Clear All" button
- **Testing**: User testing with actual RDOs
- **Iterative**: Start with basic filters, enhance based on feedback

### 3. **Data Freshness**
- **Risk**: Availability status could be stale if week changes
- **Mitigation**: Real-time subscription to week/availability changes
- **Refresh**: Add manual refresh button to grid
- **Cache**: Implement smart cache invalidation

### 4. **Breaking Existing RDO Features**
- **Risk**: New navigation could break existing Command Center
- **Mitigation**: Isolate changes, keep existing routes unchanged
- **Testing**: Verify all Phase 17.1-17.4 features still work
- **Rollback**: Prepared to revert navigation changes if issues
