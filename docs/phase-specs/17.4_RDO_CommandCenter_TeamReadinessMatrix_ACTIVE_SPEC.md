## 🏗️ Architecture Decisions

### 1. **Database Optimization Strategy**
*Do not do N+1 queries in the frontend.* To get the readiness data cleanly, we have two options:

**Option A: Supabase Database Function (RPC)**
- Create a PostgreSQL function that returns aggregated data for all clubs mapped to the RDO
- Single database call, maximum performance
- Complex SQL logic hidden from frontend

**Option B: Optimized Joined Query in Frontend**
- Single Supabase query with multiple joins and aggregations
- More complex frontend logic but keeps logic in TypeScript
- Easier to debug and modify

**Decision:** Implement **Option B** first for simplicity and debuggability. If performance becomes an issue, migrate to Option A.

### 2. **Data Aggregation Query Design**
The query needs to fetch for *all* clubs mapped to the RDO:
1. **Club Name** - from `clubs` table
2. **Current Roster Size** - Count of players where `is_active = true`
3. **Weekly Availability** - For the "current week" (most recent week created for that club), count how many `availability_responses` exist
4. **Selection Status** - Check the `team_selections` table for the current week to see if a team is drafted or finalized

### 3. **UI Component Structure**
- **Top Section**: Existing Launchpad Club Cards (unchanged)
- **New Section**: "Weekly Readiness Matrix" data table
- **Responsive Layout**: Cards above, table below, stacked on mobile
- **Loading States**: Separate loading for cards vs matrix data

### 4. **Custom Hook Design**
Create `useRDOReadiness` hook with:
- Role guard (`role === 'rdo'`)
- Loading and error state management
- Data transformation logic
- Automatic refresh on data changes

## 🎨 Logic Implementation

### Step 1: Database Query Optimization

**Aggregated Readiness Query:**
```typescript
const { data, error } = await supabase
  .from('rdo_club_access')
  .select(`
    club_id,
    clubs (
      id,
      name,
      club_settings (logo_url),
      players (
        id,
        is_active
      ),
      weeks (
        id,
        start_date,
        availability_responses (
          id,
          player_id
        ),
        team_selections (
          id,
          player_id
        )
      )
    )
  `)
  .eq('user_id', user.id)
  .order('clubs(name)')
```

**Data Transformation Logic:**
- Calculate roster size: `club.players.filter(p => p.is_active).length`
- Find current week: `club.weeks.sort((a,b) => new Date(b.start_date) - new Date(a.start_date))[0]`
- Calculate availability %: `(currentWeek.availability_responses.length / activeRoster) * 100`
- Determine selection status: Based on `team_selections` existence

### Step 3: Custom Hook Implementation

**`useRDOReadiness.ts` Structure:**
```typescript
export interface ClubReadiness {
  clubId: string;
  clubName: string;
  logoUrl?: string;
  rosterSize: number;
  availabilityPercent: number;
  selectionStatus: 'missing' | 'draft' | 'locked';
  currentWeekId?: string;
}

export function useRDOReadiness() {
  const { user, role } = useAuth();
  const [data, setData] = useState<ClubReadiness[]>([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<string | null>(null);

  useEffect(() => {
    // Role guard
    if (role !== 'rdo' || !user) {
      setData([]);
      setLoading(false);
      return;
    }

    // Race condition protection
    let ignore = false;

    async function fetchReadinessData() {
      setLoading(true);
      setError(null);

      try {
        // Aggregated query implementation
        const readinessData = await fetchAggregatedReadiness(user.id);
        
        if (!ignore) {
          setData(readinessData);
          setLoading(false);
        }
      } catch (err) {
        if (!ignore) {
          setError(err instanceof Error ? err.message : 'Failed to fetch readiness data');
          setLoading(false);
        }
      }
    }

    fetchReadinessData();

    return () => { ignore = true };
  }, [user, role]);

  return { data, loading, error };
}
```
