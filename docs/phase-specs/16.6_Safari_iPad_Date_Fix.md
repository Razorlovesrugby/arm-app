# ACTIVE_SPEC: 16.6 - Safari iPad Date Selection Fix

### 🎯 Why
When opening a new week on Safari iPad, users cannot select dates in the "Create New Week" form. The native `<input type="date">` has poor Safari iPad compatibility (touch/click handling issues, date picker rendering problems), while it works fine on Chrome. This blocks users from creating new match weeks on iPad devices, which is a critical mobile use case for coaches and team managers.

### 🧠 Context Gathering (MANDATORY FIRST STEP)
Before writing any code, you MUST examine the current date input implementation:

1. **File Location**: `/workspaces/arm-app/src/pages/Weeks.tsx`
2. **Component**: `CreateWeekForm` (lines 217-247)
3. **Current Implementation**: Uses native `<input type="date">` with `YYYY-MM-DD` format
4. **Browser Compatibility Issue**: Works on Chrome, fails on Safari iPad
5. **Date Helpers**: `getCurrentMonday()` and `getCurrentSunday()` functions provide default dates

**Safari iPad Specific Issues:**
- Touch/click handling problems with native date picker
- Date format parsing inconsistencies
- Visual rendering issues with date picker UI
- Keyboard interaction problems on touch devices

### 🏗️ Architecture Decisions

1. **Replace Native Date Inputs**: Use `<input type="text">` with `YYYY-MM-DD` format validation instead of native `<input type="date">`
2. **Mobile-First Approach**: Ensure the solution works well on touch devices (iPad) with clear visual cues
3. **Backward Compatibility**: Maintain existing date format (`YYYY-MM-DD`) and all validation logic
4. **Minimal Changes**: Avoid adding new dependencies or complex date picker libraries
5. **User Experience**: Add placeholder text and format hints to guide users

**2. Enhanced Date Validation**
Add format validation to ensure `YYYY-MM-DD` format:

```typescript
function isValidDateFormat(dateStr: string): boolean {
  const regex = /^\d{4}-\d{2}-\d{2}$/;
  if (!regex.test(dateStr)) return false;
  
  const date = new Date(dateStr + 'T00:00:00');
  return date instanceof Date && !isNaN(date.getTime());
}
```
