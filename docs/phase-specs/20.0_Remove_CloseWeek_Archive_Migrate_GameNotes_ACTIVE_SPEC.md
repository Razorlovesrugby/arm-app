## 🎯 Why This Matters

The app has **dead code** and **unused functionality** that adds maintenance burden and confuses the codebase:

1. **`close_week` RPC** — A database function that closes weeks and snapshots player data. **Never called from the UI** — the `closeWeek()` method in `useWeeks.ts` exists but has no button, trigger, or user-facing action anywhere in the app.

2. **`Archive.tsx` page** — 711 lines of code with two sub-tabs (Archive + Search). **Not routed in App.tsx, not linked in Sidebar** — completely unreachable by users.

3. **`archive_game_notes` table** — Stores per-player post-match notes. Only populated by `close_week` (which never runs). The data model is sound but the access pattern is wrong — coaches should be able to add notes directly from the Results page.

4. **`players.last_played_date` / `players.last_played_team`** — Currently only updated by `close_week`. Should instead be triggered automatically when a coach enters a match result for a week where a player was selected.

### What We're Doing

| Action | Reason |
|---|---|
| 🗑️ Delete `close_week` RPC | Dead code — never called |

## 🏗️ Architecture Decisions

### 1. **New `player_match_notes` Table (Separate from match_events)**
- **Why separate**: Clean separation of concerns — match events track scoring, notes track coach observations
- **Keyed by**: `(week_id, week_team_id, player_id)` — one note per player per match
- **No snapshots**: Unlike `archive_game_notes`, we don't snapshot player name/type/position. The join to `players` is live, so notes always show current player info.
- **RLS**: Same club-scoped pattern as all other tenant tables

### 2. **Notes UI Pattern (Expandable Rows — Like Old Archive)**
- In `ResultDetail.tsx`, each team's squad list gets a **"Notes" button** per player
- Tapping it expands a textarea inline (same pattern as the old Archive's `GameNoteRow`)
- Auto-saves with debounce (1.5s after typing stops)
- No separate modal/sheet — keeps it lightweight and accessible

### 3. **`last_played` Trigger (Score-Based, Not Close-Based)**
- **Trigger event**: `UPDATE` on `week_teams` table when `score_for` changes from `NULL` to a value
- **Action**: For the updated `week_team`, find all players in `team_selections.player_order` for that week, update their `last_played_date` and `last_played_team`
- **Why**: This ties "last played" to actual match results, not to an arbitrary "close week" action
- **Idempotent**: Re-running the same score update doesn't double-count

### 4. **No More "Closed" Status**
- The `weeks.status` column stays (it's used in `useWeeks` filtering: `openWeeks` vs `closedWeeks`)
- But nothing in the app will ever set `status = 'Closed'` anymore
- All weeks remain "Open" forever — the Results page already handles this by checking for scores
- **Future consideration**: Could add an "Archive" filter that hides weeks older than N months, but that's out of scope

```sql
-- Migration 030: Create player_match_notes table
-- Phase 19.2 — Per-player match notes in Results tab

CREATE TABLE IF NOT EXISTS player_match_notes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  week_id UUID REFERENCES weeks(id) ON DELETE CASCADE NOT NULL,
  week_team_id UUID REFERENCES week_teams(id) ON DELETE CASCADE NOT NULL,
  player_id UUID REFERENCES players(id) ON DELETE CASCADE NOT NULL,
  notes TEXT,
  club_id UUID REFERENCES clubs(id) NOT NULL,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(week_id, week_team_id, player_id)
);

-- Enable RLS
ALTER TABLE player_match_notes ENABLE ROW LEVEL SECURITY;

-- Coach + RDO access (club-scoped)
CREATE POLICY player_match_notes_coach_all ON player_match_notes
  FOR ALL TO authenticated
  USING (
    club_id IN (SELECT club_id FROM profiles WHERE id = auth.uid())
    OR
    EXISTS (
      SELECT 1 FROM rdo_club_access
      WHERE user_id = auth.uid() AND club_id = player_match_notes.club_id
    )
  )
  WITH CHECK (
    club_id IN (SELECT club_id FROM profiles WHERE id = auth.uid())
    OR
    EXISTS (
      SELECT 1 FROM rdo_club_access
      WHERE user_id = auth.uid() AND club_id = player_match_notes.club_id
    )
  );

-- Service role bypass
CREATE POLICY player_match_notes_service_role ON player_match_notes
  FOR ALL TO service_role USING (true) WITH CHECK (true);

-- Auto-update updated_at
CREATE OR REPLACE FUNCTION update_player_match_notes_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER player_match_notes_updated_at
  BEFORE UPDATE ON player_match_notes
  FOR EACH ROW
  EXECUTE FUNCTION update_player_match_notes_updated_at();

-- Index for fast lookups
CREATE INDEX IF NOT EXISTS idx_player_match_notes_week_team
  ON player_match_notes(week_id, week_team_id);
CREATE INDEX IF NOT EXISTS idx_player_match_notes_player
  ON player_match_notes(player_id);
```

### Phase 3: `last_played` Trigger (Migration 031)

**New file**: `supabase/migrations/031_last_played_trigger.sql`

```sql
-- Migration 031: Auto-update players.last_played_date and last_played_team
-- Phase 19.2 — Triggered when a match result (score) is entered

CREATE OR REPLACE FUNCTION update_last_played_on_score()
RETURNS TRIGGER
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
DECLARE
  v_week_start DATE;
  v_team_name TEXT;
  v_player_id UUID;
BEGIN
  -- Only fire when score_for changes from NULL to a value (result entered)
  IF OLD.score_for IS NOT NULL OR NEW.score_for IS NULL THEN
    RETURN NEW;
  END IF;

  -- Get week start date and team name
  SELECT w.start_date INTO v_week_start
  FROM weeks w
  WHERE w.id = NEW.week_id;

  SELECT wt.team_name INTO v_team_name
  FROM week_teams wt
  WHERE wt.id = NEW.id;

  IF NOT FOUND THEN
    RETURN NEW;
  END IF;

  -- Update all players in this team's selection for this week
  FOR v_player_id IN
    SELECT elem::uuid
    FROM team_selections ts
    CROSS JOIN LATERAL jsonb_array_elements_text(ts.player_order) AS elem
    WHERE ts.week_id = NEW.week_id
      AND ts.week_team_id = NEW.id
      AND elem IS NOT NULL
  LOOP
    UPDATE players
    SET
      last_played_date = v_week_start,
      last_played_team = v_team_name
    WHERE id = v_player_id;
  END LOOP;

  RETURN NEW;
END;
$$;

-- Attach trigger to week_teams
CREATE TRIGGER trigger_update_last_played_on_score
  AFTER UPDATE OF score_for ON week_teams
  FOR EACH ROW
  EXECUTE FUNCTION update_last_played_on_score();
```

```typescript
// New component inside ResultDetail.tsx (or extracted to its own file)

interface PlayerNotesProps {
  playerId: string
  playerName: string
  weekId: string
  weekTeamId: string
  clubId: string
  initialNotes: string | null
}

function PlayerNotesRow({ playerId, playerName, weekId, weekTeamId, clubId, initialNotes }: PlayerNotesProps) {
  const [expanded, setExpanded] = useState(false)
  const [notes, setNotes] = useState(initialNotes ?? '')
  const [saving, setSaving] = useState(false)
  const debounceRef = useRef<ReturnType<typeof setTimeout> | null>(null)

  // Sync if external data changes
  useEffect(() => {
    setNotes(initialNotes ?? '')
  }, [initialNotes])

  function handleChange(val: string) {
    setNotes(val)
    if (debounceRef.current) clearTimeout(debounceRef.current)
    debounceRef.current = setTimeout(async () => {
      setSaving(true)
      await supabase
        .from('player_match_notes')
        .upsert({
          week_id: weekId,
          week_team_id: weekTeamId,
          player_id: playerId,
          notes: val.trim() || null,
          club_id: clubId,
        }, { onConflict: 'week_id, week_team_id, player_id' })
      setSaving(false)
    }, 1500)
  }

  return (
    <div>
      <div
        onClick={() => setExpanded(e => !e)}
        style={{ display: 'flex', alignItems: 'center', gap: 8, padding: '8px 0', cursor: 'pointer' }}
      >
        <span style={{ fontSize: 14, fontWeight: 600, color: '#111827', flex: 1 }}>
          {playerName}
        </span>
        <span style={{ fontSize: 12, color: '#6B21A8', fontWeight: 600 }}>
          {expanded ? '▲ Hide Notes' : '📝 Notes'}
        </span>
      </div>
      {expanded && (
        <div style={{ paddingLeft: 0, paddingBottom: 8 }}>
          <textarea
            value={notes}
            onChange={e => handleChange(e.target.value)}
            placeholder="Post-match notes for this player…"
            rows={3}
            style={{
              width: '100%', boxSizing: 'border-box',
              padding: '10px 12px',
              border: '1px solid #E5E7EB',
              borderRadius: 8,
              fontSize: 14, color: '#111827',
              background: '#FFFFFF',
              resize: 'vertical',
              outline: 'none',
              fontFamily: 'inherit',
              lineHeight: 1.5,
            }}
            onFocus={e => (e.target.style.borderColor = '#6B21A8')}
            onBlur={e  => (e.target.style.borderColor = '#E5E7EB')}
          />
          {saving && (
            <p style={{ margin: '4px 0 0', fontSize: 11, color: '#9CA3AF' }}>Saving…</p>
          )}
        </div>
      )}
    </div>
  )
}
```

```typescript
// Load player_match_notes for all teams in this week
const [playerNotes, setPlayerNotes] = useState<Record<string, string | null>>({}) // keyed by player_id

useEffect(() => {
  if (!week) return
  const teamIds = week.week_teams.filter(t => t.is_active !== false).map(t => t.id)
  if (teamIds.length === 0) return

  async function loadNotes() {
    const { data } = await supabase
      .from('player_match_notes')
      .select('*')
      .in('week_team_id', teamIds)
      .eq('week_id', week.id)

    const notesMap: Record<string, string | null> = {}
    for (const row of (data ?? []) as PlayerMatchNote[]) {
      notesMap[row.player_id] = row.notes
    }
    setPlayerNotes(notesMap)
  }
  loadNotes()
}, [week?.id])
```

## ⚠️ Risks & Mitigations

### Risk 1: `last_played` Trigger Fires on Every Score Update
- **Likelihood**: Medium
- **Impact**: Low (just re-runs the same UPDATE)
- **Mitigation**: Trigger checks `OLD.score_for IS NULL AND NEW.score_for IS NOT NULL` — only fires on first score entry
- **Edge case**: If score is cleared and re-entered, it fires again — acceptable (idempotent UPDATE)

### Risk 2: `player_match_notes` RLS Blocks Coach Access
- **Likelihood**: Low
- **Impact**: High (coaches can't save notes)
- **Mitigation**: Follows exact same RLS pattern as all other tenant tables — tested pattern

### Risk 3: Debounce Saves Lost on Navigation
- **Likelihood**: Medium
- **Impact**: Low (notes saved in textarea state, user can re-type)
- **Mitigation**: Could add `beforeunload` warning, but overkill for this use case

### Risk 4: Existing `archive_game_notes` Data Loss
- **Likelihood**: Certain (we're dropping the table)
- **Impact**: Low (table is empty — close_week never ran)
- **Mitigation**: If any data exists, we could migrate it to `player_match_notes` before dropping

## 🏁 Conclusion

This spec cleans up dead code (close_week RPC, Archive page, archive_game_notes table) and replaces it with a cleaner, more intuitive workflow:

1. **Per-player match notes** live in a dedicated `player_match_notes` table, accessible directly from the Results page
2. **`last_played` fields** update automatically when a coach enters a match score, not via a separate "close week" action
3. **No more dead code** — the Archive page, close_week RPC, and CloseWeekWarning type are all removed

The result is a leaner codebase and a more intuitive UX: coaches enter scores, add notes per player, and the system handles the rest.
