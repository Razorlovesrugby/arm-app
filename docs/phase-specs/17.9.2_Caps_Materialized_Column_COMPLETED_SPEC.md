# COMPLETED SPEC: Phase 17.9.2 - Caps Materialized Column

## Implementation Date: 2026-04-20
## Status: ✅ Complete

## 1. Context & Objective

Phase 17.9.2 implements a permanent solution to the caps calculation issue by replacing the async compute-on-read RPC pattern with a materialized `total_caps` column on the `players` table. Despite Phase 17.9.1 fixing the RPC with `SECURITY DEFINER`, the root issue of PostgREST/RLS interaction in deployed environments remained unresolved.

**Primary Objective:** Replace the async `calculate_player_caps` RPC call with an instant-read materialized column, kept in sync by database triggers, eliminating frontend async complexity and RLS interaction issues.

## 2. Problem Analysis

### Issues Identified
1. **Inconsistent RPC Results:** The `calculate_player_caps` RPC returned correct values in Supabase SQL console but returned 0 when called via Supabase JS client on deployed frontend.
2. **RLS/PostgREST Interaction:** Root cause was unresolved PostgREST/RLS interaction that made debugging complex and unreliable.
3. **Frontend Complexity:** Async fetch pattern in `PlayerOverlay.tsx` added loading states, error handling, and race conditions.
4. **Performance:** RPC calls on every overlay open created unnecessary database load.

### Root Causes
- PostgREST translation of RPC calls through RLS policies inconsistent in production
- Complex multi-tenant RLS policies interfering with function execution
- Async pattern introduced UI complexity and potential race conditions
- No caching of cap counts led to repeated expensive calculations

## 3. Solution Architecture

### 3.1 Materialized Column Pattern
```sql
-- Migration 029_phase_17_9_2_caps_materialized.sql
-- 1. Add column
ALTER TABLE players ADD COLUMN IF NOT EXISTS total_caps INT NOT NULL DEFAULT 0;

-- 2. Refresh function (reuses existing cap logic)
CREATE OR REPLACE FUNCTION refresh_player_caps(p_player_id UUID)
RETURNS VOID AS $$
DECLARE
  v_historical INT;
  v_match      INT;
BEGIN
  SELECT COALESCE(historical_caps, 0) INTO v_historical
  FROM players WHERE id = p_player_id;

  SELECT COUNT(DISTINCT ts.week_id)::INT INTO v_match
  FROM team_selections ts
  JOIN week_teams wt ON ts.week_team_id = wt.id
  WHERE EXISTS (
    SELECT 1
    FROM jsonb_array_elements_text(ts.player_order) WITH ORDINALITY AS elem(player_id, idx)
    WHERE elem.player_id::UUID = p_player_id
      AND elem.idx BETWEEN 1 AND 23
  )
  AND (wt.score_for IS NOT NULL OR wt.score_against IS NOT NULL);

  UPDATE players
  SET total_caps = COALESCE(v_historical, 0) + COALESCE(v_match, 0)
  WHERE id = p_player_id;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

**Key Decisions:**
1. **Materialized column, not a view**: `players.total_caps INT NOT NULL DEFAULT 0` lives on the players row, reads instantly
2. **Trigger-maintained**: Two triggers keep it in sync: one on `team_selections` (player_order changes), one on `week_teams` (score changes)
3. **Single refresh function**: `refresh_player_caps(UUID)` contains all calculation logic, called by both triggers and used for backfill
4. **No `calculate_player_caps` RPC changes**: Existing RPC left intact for backward compatibility
5. **Frontend zero-async**: `PlayerOverlay` reads `player.total_caps` directly

### 3.2 Trigger Architecture

#### 3.2.1 Team Selections Trigger
```sql
CREATE OR REPLACE FUNCTION trg_caps_on_selection_change()
RETURNS TRIGGER AS $$
DECLARE pid TEXT;
BEGIN
  IF TG_OP IN ('DELETE', 'UPDATE') THEN
    FOR pid IN SELECT jsonb_array_elements_text(OLD.player_order) LOOP
      IF pid IS NOT NULL THEN PERFORM refresh_player_caps(pid::UUID); END IF;
    END LOOP;
  END IF;
  IF TG_OP IN ('INSERT', 'UPDATE') THEN
    FOR pid IN SELECT jsonb_array_elements_text(NEW.player_order) LOOP
      IF pid IS NOT NULL THEN PERFORM refresh_player_caps(pid::UUID); END IF;
    END LOOP;
  END IF;
  RETURN NULL;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

#### 3.2.2 Week Teams Score Trigger
```sql
CREATE OR REPLACE FUNCTION trg_caps_on_score_change()
RETURNS TRIGGER AS $$
DECLARE pid TEXT;
BEGIN
  IF (OLD.score_for IS DISTINCT FROM NEW.score_for) OR
     (OLD.score_against IS DISTINCT FROM NEW.score_against) THEN
    FOR pid IN
      SELECT DISTINCT jsonb_array_elements_text(ts.player_order)
      FROM team_selections ts
      WHERE ts.week_team_id = NEW.id
    LOOP
      IF pid IS NOT NULL THEN PERFORM refresh_player_caps(pid::UUID); END IF;
    END LOOP;
  END IF;
  RETURN NULL;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

### 5.2 Edge Cases Handled
- **Null slots in `player_order`**: Trigger loops skip `NULL` pid values
- **Player in multiple scored weeks**: `COUNT(DISTINCT ts.week_id)` deduplicates
- **`historical_caps` NULL**: `COALESCE` in refresh function handles it
- **RLS on `players`**: `SECURITY DEFINER` on all new functions bypasses it
- **Multiple triggers firing**: Deduplication in score trigger with `DISTINCT`

## 7. Performance Considerations

### 7.1 Read Performance
- **Instant reads**: Column value available immediately, no async fetch
- **No network latency**: Data comes with player object in existing queries
- **Reduced database load**: Eliminates RPC calls on every overlay open

### 7.2 Write Performance
- **Trigger overhead**: Minimal - refreshes only affected players
- **Bulk operations**: Triggers handle batch updates efficiently
- **Backfill**: One-time operation, can be run during maintenance

### 7.3 Storage Impact
- **Additional column**: 4 bytes per player (INT)
- **Negligible overhead**: ~4KB for 1000 players
